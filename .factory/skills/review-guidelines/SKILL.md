# 团队评审指引

## 输出语言（最重要）

- 所有评审输出——摘要、行内评论、修复建议说明——一律使用简体中文。
- 代码标识符、文件路径、API/库名、专业术语保留英文原文。
- 严重度前缀保留 [P1]/[P2]/[P3] 格式。

## 降噪规则

- 只报告高置信度的真实缺陷：逻辑错误、边界条件、空值解引用、并发竞态、资源泄漏、安全漏洞、数据一致性问题。
- 不评论代码风格、命名偏好、纯架构观点和无实质收益的微优化。
- 跳过自动生成的代码：src/interface/ 目录、*.generated.*、openapi/swagger 生成物、lock 文件。

## 输出格式规范

### 行内评论（每个问题一条）

第一行固定为 `[P0] 一句话问题标题`（或 [P1]/[P2]，方括号前缀必须在 body 最前面），随后两部分：

- 问题与影响：哪里错了、什么场景触发、后果是什么（让研发不用追问就能定位）。
- 修复方案：给出具体改法；高置信度的修复必须附 suggestion 代码块；不确定的用提问式表述并注明不确定性。

严重度口径：

- P0 = 必然或大概率的运行时故障/数据错误/安全漏洞（崩溃、资损、越权、数据不一致）。
- P1 = 特定条件下的真实缺陷或高风险写法（边界条件、并发竞态、资源泄漏、错误被吞掉）。
- P2 = 不影响当前正确性但值得修的实质问题（性能隐患、可靠性弱化）。

### 摘要评论

按顺序包含：本轮发现统计（P0/P1/P2 数量）→ 逐条「文件:行 — 问题 — 修复方向」→ 建议的修复顺序。

**禁止输出任何评分、星级或"可否合并"的结论**——评分与合并门禁由系统脚本单独计算并发布（`droid-review/gate` 状态与评分卡评论）。你的职责是发现真实问题并给出修复路径。

## 复评与放行规则

收到"复评"类请求（如 `@droid 复评`）时，先逐条核对已有评论线程，再生成本轮 candidates：

- 代码已修复，或研发在线程中的解释成立（确属误报/预期行为/已有外部保障）→ 不再报该问题。
- 线程被 resolve 但代码并未修复、且无合理解释 → **必须重新报出**，不要因为"已有相同评论"而去重跳过。
- 本次新 push 引入的问题照常报。

判断研发解释是否成立时，以代码事实为准：宁可重报，不可放过真实缺陷。

## 技术栈背景

- 后端仓库：Java Spring Boot，生产分支 master。
- 前端仓库：React / UmiJS / Taro 微信小程序，生产分支 prod；TypeScript 项目关注类型安全。
- test 和 release 分支可直接推送并自动部署，评审重点放在进入生产分支的 PR。
